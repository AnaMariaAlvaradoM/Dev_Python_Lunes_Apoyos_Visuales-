De "crear y listar" al ciclo de vida completo de un dato

Hoy tu API aprende a pedir un producto específico, filtrarlos, actualizarlos y borrarlos. Vamos a completar el CRUD.

📞

Imagina que tu API ya está en producción y un cliente llama molesto:
"Compré un producto, pero subió de precio y necesito que lo corrijan. Y ese otro que ya no venden, ¿por qué sigue apareciendo en la app?"

Abres tu código de la clase pasada y notas algo incómodo: solo sabes crear productos y listarlos todos. No tienes forma de:

  • Pedir un producto específico (el que el cliente compró)
  • Cambiarle el precio a ese producto
  • Borrar el que ya no se vende

¿Cómo le digo al servidor "de todos los productos, quiero trabajar con ESTE en particular", y luego "actualízalo" o "bórralo"?

Vamos a completar el ciclo de vida de un dato: nace, se consulta, se modifica y se elimina.

La clase pasada lograste tres cosas, y todas reaparecen hoy:

?Si te dijera "tráeme el segundo producto que creaste", ¿cómo se lo pedirías al servidor? ¿Tus productos tenían forma de identificarse de manera única?▾
💭 Lo que solemos pensar

"Por el nombre" o "no sé, no tenía número".

✓ La respuesta

No tenía forma. Guardábamos nombre, precio y stock, pero ningún identificador único. Si dos productos se llaman "Mouse", no los distingues. Pedir "por nombre" es frágil: los nombres se repiten y cambian. Hoy resolvemos esto primero, porque todo lo demás depende de ello.

El concepto que nos faltaba: la identidad

Tú tienes un número de cédula. Puede haber muchas personas llamadas "Ana", pero solo una tiene tu número. Ese número no se repite y no cambia aunque cambies de casa, de trabajo o de apellido.

Definición técnica

Un identificador único (ID) es un valor que distingue sin ambigüedad a un registro de todos los demás dentro de una colección. Por convención suele ser un número entero que crece (1, 2, 3...).

🪪

En la vida real = tu cédula. En APIs y bases de datos pasa igual: cada registro necesita un ID que nunca se repita. Así, cuando alguien dice "el producto 5", no hay ninguna duda de cuál es.

🆔 Identidad

Darle ID a los productos

Antes que cualquier verbo nuevo, cada producto necesita su número único. De ahí en adelante, todo se vuelve posible.

Cómo guardábamos antes

La clase pasada, dentro del POST, guardábamos el producto así:

python 
productos_db.append(producto)

Ese producto era el objeto Pydantic crudo. Funcionaba, pero hoy nos topamos con dos límites: no tiene ID y es difícil de modificar (algo que el PUT necesitará).

🔄

Evolución natural, no corrección: ahora guardaremos un diccionario que incluye su propio id. El objeto Pydantic sigue siendo el rey para validar la entrada; el diccionario es mejor para almacenar y modificar. Cada herramienta en su rol.

?Si voy creando productos uno tras otro y quiero que sean 1, 2, 3... ¿qué necesito mantener "vivo" fuera de los endpoints para saber qué número me toca asignar?▾
💭 Lo que solemos pensar

"Un contador" o "una variable que sume".

✓ La respuesta

Exacto: un contador que viva fuera de las funciones y recuerde el último ID entregado. Como vive fuera, su valor sobrevive entre una petición y la siguiente, así que nunca repite número.

Paso 1 · Agregar el contador

📍 app/main.py — justo debajo de productos_db = [], antes de los endpoints
python 
# Contador que recuerda cuál es el próximo ID a asignar.
# Vive fuera de los endpoints para "recordar" entre una petición y otra.
contador_id = 1

contador_id vive en el nivel del módulo (sin indentación, fuera de toda función), como productos_db. Las variables dentro de una función nacen y mueren con cada llamada; las del nivel del módulo persisten mientras el servidor esté encendido. Empieza en 1 porque queremos que el primer producto sea el número 1.

Paso 2 · Agregar la categoría al schema

Hoy también filtraremos productos. Pero para filtrar por una característica, esa característica debe existir en los datos. Por eso sembramos la categoría hoy: la cosecharemos en los filtros.

📍 app/main.py — en tu schema ProductoCreate
python 
class ProductoCreate(BaseModel):
    nombre: str
    precio: float
    stock: int
    categoria: str   # ← línea nueva

Con esa única línea, Pydantic ahora exige que todo producto traiga su categoría. Sin if, sin chequeo manual: declaras el campo y Pydantic se encarga. Declaras qué quieres, no cómo validarlo.

?Si un compañero prueba el POST con el JSON viejo (solo nombre, precio y stock), ¿qué pasará ahora que categoria es obligatorio?▾
💭 Lo que solemos pensar

"Funcionará igual" o "dará error".

✓ La respuesta

Dará un error 422, el mismo de validación de la clase pasada. Como falta categoria, Pydantic rechaza la petición y señala el campo faltante. No es un problema: es la prueba de que el schema manda.

Paso 3 · Reescribir el POST para asignar ID

📍 app/main.py — reemplaza tu @app.post("/productos") completo
python 
@app.post("/productos")
def crear_producto(producto: ProductoCreate):
    global contador_id
    nuevo = {
        "id": contador_id,
        "nombre": producto.nombre,
        "precio": producto.precio,
        "stock": producto.stock,
        "categoria": producto.categoria
    }
    productos_db.append(nuevo)
    contador_id += 1
    return {"mensaje": "Producto creado exitosamente", "producto": nuevo}

¿Qué hace cada parte?

  • global contador_id → le dice a Python: "el contador que modifico aquí es el de afuera, no uno local nuevo". Sin esto, al hacer += 1 Python crearía un contador local fantasma y todos los productos tendrían id 1.
  • El diccionario nuevo → guarda el id actual del contador y copia los campos del objeto validado (producto.nombre, etc.). Traducimos del objeto Pydantic a un diccionario almacenable, agregándole identidad.
  • productos_db.append(nuevo) → mismo .append() de siempre; ahora guarda un diccionario con id.
  • contador_id += 1 → sube el contador para que el próximo producto reciba el siguiente número. Sin esto, todos colisionarían en id 1.
  • El return devuelve el producto ya con su id, para que el cliente sepa con qué número quedó.

Paso 4 · El GET de la lista ya muestra el id

📍 app/main.py — ya existe; por ahora déjalo igual
python 
@app.get("/productos")

    return {"total": len(productos_db), "productos": productos_db}

Como la lista ahora contiene diccionarios con id, este GET ya los muestra automáticamente. En la sección de filtros lo ampliaremos.

"El 4, sigue contando".

✓ La respuesta

📍 Parámetros de ruta

Pedir UN producto específico

Ya cada producto tiene identidad. Ahora usémosla para pedir uno solo.

Definición técnica

Un parámetro de ruta (path parameter) es una porción variable de la URL que identifica un recurso concreto dentro de una colección. Es obligatorio y forma parte de la dirección misma.

rutas 
/productos        → todos los productos
/productos/5      → el producto específico con id 5
📚

En la vida real: /productos es pedirle al bibliotecario "el catálogo completo". /productos/5 es "tráeme exactamente el libro del estante 5". El número es parte de la dirección: cambias el número, cambias el libro.

?Si /productos ya da la lista completa, ¿para qué pedir un endpoint aparte que me dé uno solo?▾
💭 Lo que solemos pensar

"Para no traer todo" o "es más directo".

✓ La respuesta

Por eficiencia y claridad. Con 50.000 productos, traerlos todos para usar uno es absurdo: gasta datos, memoria y tiempo. Pedir /productos/5 es quirúrgico. Y además expresa la intención con precisión: la URL misma dice "quiero el producto 5". Esa es la filosofía REST.

Antes de escribir: ¿qué es raise?

Definición técnica

raise es una palabra clave de Python que lanza (provoca) una excepción a propósito. Normalmente las excepciones saltan solas cuando algo falla; con raise eres tú quien decide provocarla, porque detectaste una situación que no debe seguir.

💂

El guardia de la lista: revisa nombre por nombre. Si te encuentra, te deja pasar (return). Si llega al final y no estás, levanta la mano y anuncia en voz alta "¡no está en la lista!" — eso es raise: una interrupción deliberada para que el sistema reaccione.

return = "todo bien, aquí está". raise = "detente, este es el problema". Solo uno se ejecuta por llamada.

Paso 1 · Importar HTTPException

📍 app/main.py — primera línea, la del import de FastAPI
python 
from fastapi import FastAPI, HTTPException

La misma línea de siempre, ahora trae dos cosas separadas por coma. HTTPException es "una excepción con forma de respuesta HTTP": cuando la lanzas con raise, FastAPI la convierte en un error limpio con su código y mensaje, en vez de que el servidor se caiga feo.

Paso 2 · El endpoint GET por ID

📍 app/main.py — endpoint nuevo, después del GET de la lista
python 
@app.get("/productos/{producto_id}")
def obtener_producto(producto_id: int):
    for p in productos_db:
        if p["id"] == producto_id:
            return p
    raise HTTPException(status_code=404, detail="Producto no encontrado")

Línea por línea

  • "/productos/{producto_id}" → las llaves marcan la parte variable. Lo que el cliente escriba ahí, FastAPI lo captura y lo pasa a la función.
  • producto_id: int → el type hint hace que FastAPI convierta y valide que sea entero. Si alguien escribe /productos/abc, responde 422 automáticamente.
  • for p in productos_db → recorremos la lista producto por producto.
  • if p["id"] == producto_id: return p → si el id coincide, lo devolvemos. Ese return corta el bucle: no buscamos algo ya encontrado.
  • raise HTTPException(...) → fíjate en la indentación: está alineado con el for, así que solo corre si el bucle terminó sin encontrar nada. Lanza el 404 con su mensaje.

Los códigos de estado HTTP

CódigoSignificaCuándo aparece
200OKTodo bien, aquí está lo pedido
404Not FoundPediste un recurso que no existe
422Unprocessable EntityDatos mal formados (tipo incorrecto, falta un campo)
🍽️

Como un mesero: 200 = "aquí está su plato". 404 = "ese plato no existe en el menú". 422 = "su orden está incompleta". Cada respuesta dice no solo qué pasó, sino de qué tipo fue.

?Si pides /productos/999 y no existe, ¿qué es mejor: devolver una lista vacía, o decir claramente "no encontrado" con un 404?▾
💭 Lo que solemos pensar

"Que diga no encontrado".

✓ La respuesta

El 404. Devolver vacío sería mentir sutilmente: diría "la búsqueda funcionó pero no hay nada", cuando la verdad es "no existe". Con un 404 honesto, el frontend muestra "Producto no disponible" en vez de una pantalla en blanco confusa. Decir la verdad con el código correcto separa una API amateur de una profesional.

🔎 Parámetros de query

Filtrar la colección

No un producto, ni todos: un subconjunto que cumple condiciones. "Todos los de tecnología", "los que cuestan menos de 50 mil".

rutas 
/productos?categoria=tecnologia
/productos?precio_max=50
/productos?categoria=tecnologia&precio_max=50
  • El ? marca dónde empiezan los filtros.
  • Cada filtro es un par nombre=valor.
  • El & une varios filtros.
Ruta · /productos/5

Identifica CUÁL recurso. Obligatorio. Parte de la dirección. Apunta a uno.

Query · ?categoria=x

Filtra una colección. Opcional. Va tras el ?. Puede devolver muchos, pocos o ninguno.

🚕

El taxi: la ruta es darle la dirección exacta de una casa ("Calle 5 #23") → un lugar único. El query es decir "llévame a un restaurante italiano y económico" → condiciones que filtran entre muchas opciones. Una dirección apunta; un filtro tamiza.

?¿Por qué la categoría va como ?categoria=tecnologia (query) y no como /productos/tecnologia (ruta), si ambos viajan en la URL?▾
💭 Lo que solemos pensar

"No sé, se ven parecidos".

✓ La respuesta

Significan cosas distintas. /productos/5 apunta a un recurso único (no hay otro con id 5). ?categoria=tecnologia filtra la colección: puede devolver 2, 10 o 0. La ruta responde a "¿cuál?", el query a "¿con qué condiciones?". REST le da a cada uno un trabajo claro, y por eso las APIs REST son predecibles.

El concepto: parámetro opcional

Si pido /productos a secas quiero todos. Si agrego ?categoria=tecnologia quiero solo esos. El mismo endpoint atiende ambos casos. ¿Cómo se dice "puede venir o no" en Python?

python 
categoria: str | None = None
  • categoria → el nombre del parámetro.
  • : str | None → la barra | se lee "o": es texto o nada (None = ausencia de valor).
  • = None → su valor por defecto. Esto es lo que lo hace opcional: si no lo mandan, vale None.

Ampliar el GET de la lista con filtros

📍 app/main.py — reemplaza tu GET /productos completo
python 
@app.get("/productos")
def listar_productos(categoria: str | None = None, precio_max: float | None = None):
    resultado = productos_db
    if categoria is not None:
        resultado = [p for p in resultado if p["categoria"].lower() == categoria.lower()]
    if precio_max is not None:
        resultado = [p for p in resultado if p["precio"] <= precio_max]
    return {"total": len(resultado), "productos": resultado}

Línea por línea

  • Los dos parámetros no van entre llaves en la ruta, así que FastAPI deduce solo que son query. Lo de las llaves es path; lo que aparece solo en la función es query.
  • resultado = productos_db → arrancamos con todos. Iremos recortando.
  • if categoria is not None → solo si llegó categoría. El list comprehension se lee: "de cada p, quédate con los que su categoría coincide". El .lower() en ambos lados trata "Tecnologia" y "tecnologia" como iguales: amable con el usuario.
  • El segundo if filtra sobre resultado, que ya puede venir filtrado por categoría. Por eso los filtros se combinan de forma natural.
Dos coladores en fila: lo que sale del primero entra al segundo. Solo lo que pasa por ambos llega al final.
?Si alguien manda los DOS filtros a la vez, ¿se aplican ambos, o solo el último gana?▾
💭 Lo que solemos pensar

"Solo el último" o "no estoy seguro".

✓ La respuesta

Se aplican ambos. El segundo if no filtra sobre la lista original: filtra sobre resultado, que ya viene recortado. Como dos coladores en fila, las condiciones se acumulan. Pides tecnología Y precio < 30, y obtienes lo que cumple las dos cosas.

Las 4 combinaciones en Swagger

json · respuestas 
# 1 · sin filtros → los 3
{ "total": 3, "productos": [ Laptop, Mouse, Cuaderno ] }

# 2 · ?categoria=tecnologia → 2 (sin el Cuaderno)
{ "total": 2, "productos": [ Laptop, Mouse ] }

# 3 · ?precio_max=30 → 2 (sin la Laptop)
{ "total": 2, "productos": [ Mouse, Cuaderno ] }

# 4 · ?categoria=tecnologia&precio_max=30 → solo el Mouse
{ "total": 1, "productos": [ Mouse ] }
🔗

Mira la URL que arma Swagger (campo "Request URL") en la prueba 4: aparece literalmente ?categoria=tecnologia&precio_max=30. Ahí está, en vivo: el ? que abre y el & que une. La teoría hecha URL real.

✏️ PUT

Actualizar un producto

Volvemos al cliente molesto: "súbanme el precio que está mal". Eso es tomar algo que ya existe y cambiarle los datos.

VerboHace
GETLee (no cambia nada)
POSTCrea algo nuevo
PUTActualiza algo que ya existe
Definición técnica

PUT reemplaza el estado de un recurso existente. Por convención, envía el recurso completo: todos sus campos, no solo el que cambia.

📋

POST vs PUT: POST es llenar una ficha en blanco para un empleado que acaba de entrar. PUT es tomar la ficha que ya existe y corregirle el cargo o el salario. La identidad se mantiene; cambian los datos.

?Para actualizar el producto 5 necesito dos cosas: CUÁL cambiar (el 5) y los datos nuevos. ¿Por dónde viaja cada una: URL o body?▾
💭 Lo que solemos pensar

"El número en la URL, los datos en el body".

✓ La respuesta

¡Exacto! Y lo bonito: ya conoces las dos piezas. El "cuál" es un parámetro de ruta (lo viste en el GET por id). Los datos nuevos van en el body validado por Pydantic (como en el POST). PUT no es nuevo desde cero: es combinar dos cosas que ya dominas.

El endpoint PUT

📍 app/main.py — endpoint nuevo, después del GET por id
python 
@app.put("/productos/{producto_id}")
def actualizar_producto(producto_id: int, datos: ProductoCreate):
    for p in productos_db:
        if p["id"] == producto_id:
            p["nombre"] = datos.nombre
            p["precio"] = datos.precio
            p["stock"] = datos.stock
            p["categoria"] = datos.categoria
            return {"mensaje": "Producto actualizado", "producto": p}
    raise HTTPException(status_code=404, detail="Producto no encontrado")

Línea por línea

  • @app.put → el verbo de actualizar. La ruta lleva {producto_id}, igual que el GET por id.
  • La función recibe las dos piezas a la vez: producto_id (de la ruta) y datos: ProductoCreate (del body). Reutilizamos el mismo schema: los campos editables son los mismos que los de crear.
  • Buscamos el producto igual que en el GET por id. Ya es tu viejo conocido.
  • Sobrescribimos campo por campo (p["nombre"] = datos.nombre...). Como p es una referencia al diccionario dentro de la lista, el cambio queda hecho en su sitio: no hay que volver a guardar.
  • El id NO aparece: no lo tocamos a propósito. La identidad se conserva.
  • Mismo patrón de 404 si no se encuentra. Reusarlo en cada endpoint hace tu API consistente.
?Cambiamos nombre, precio, stock y categoría, pero NO el id. ¿Por qué sería grave permitir que una actualización cambiara el id?▾
💭 Lo que solemos pensar

"Se confundiría" o "se perdería".

✓ La respuesta

El id es la identidad permanente, como tu cédula. Si actualizar el precio pudiera cambiar el id, perderías todas las referencias: imagina que la venta #200 apuntaba al producto 5 y de pronto pasó a ser el 8. Esa venta quedaría huérfana. La regla profesional: los datos cambian, la identidad nunca.

Prueba en Swagger

json · PUT /productos/2 
# body enviado
{ "nombre": "Mouse Gamer", "precio": 45.0, "stock": 80, "categoria": "tecnologia" }

# respuesta → conserva el id 2
{
  "mensaje": "Producto actualizado",
  "producto": { "id": 2, "nombre": "Mouse Gamer", "precio": 45.0, "stock": 80, "categoria": "tecnologia" }
}
🗑️ DELETE

Borrar un producto

Última queja del cliente: "ese producto que ya no venden, quítenlo de la app". Eso es eliminar un recurso para siempre.

Definición técnica

DELETE elimina un recurso identificado por su URL. No necesita body: para borrar basta con saber cuál recurso eliminar.

🗂️

En la vida real: es sacar definitivamente la ficha de un empleado que ya no trabaja. No la editas (PUT), no la consultas (GET): la retiras.

?Para borrar el producto 5, ¿qué necesito que mande el cliente: solo el id, o también todos los datos como en el PUT?▾
💭 Lo que solemos pensar

"Solo el id" o "también los datos".

✓ La respuesta

Solo el id. ¿Para qué te diría el cliente el nombre y precio de algo que vamos a destruir? Por eso DELETE, igual que el GET por id, usa solo el parámetro de ruta y no lleva body. Cada operación pide exactamente lo que necesita, ni más ni menos.

El endpoint DELETE

📍 app/main.py — último endpoint nuevo, después del PUT
python 
@app.delete("/productos/{producto_id}")
def eliminar_producto(producto_id: int):
    for p in productos_db:
        if p["id"] == producto_id:
            productos_db.remove(p)
            return {"mensaje": f"Producto {producto_id} eliminado correctamente"}
    raise HTTPException(status_code=404, detail="Producto no encontrado")

Línea por línea

  • @app.delete y la función recibe solo el producto_id. Sin body: borrar no necesita datos. Es la firma más simple del día.
  • Buscamos por id con el mismo patrón recorrer-y-comparar que ya usamos tres veces. A estas alturas se siente automático.
  • productos_db.remove(p) → la operación inversa a .append(): append mete al final, remove quita el elemento indicado. Ese producto deja de existir en memoria.
  • La f antes de las comillas hace un f-string: inserta el valor de {producto_id} dentro del texto. Si borras el 3, el mensaje dice "Producto 3 eliminado...". Sin la f, saldrían las llaves literales.
  • Mismo patrón de 404. Cuarto endpoint que termina así, y esa repetición es buena: tu API es predecible.
?Borras el producto 3 y responde "eliminado". Acto seguido vuelves a hacer DELETE /productos/3. ¿Qué pasa en el segundo intento?▾
💭 Lo que solemos pensar

"Lo borra otra vez" o "da error".

✓ La respuesta

Responde 404. La primera vez lo eliminó; en la segunda, el for recorre la lista y ya no hay ningún id 3, así que cae en el raise. Es correcto y deseable: la API dice la verdad sobre su estado actual. No puedes borrar dos veces lo mismo.

Prueba en Swagger

json · respuestas 
# DELETE /productos/3 → 200 (el f-string mete el 3)
{ "mensaje": "Producto 3 eliminado correctamente" }

# DELETE /productos/3 otra vez → 404
{ "detail": "Producto no encontrado" }
🧩 Síntesis

El CRUD completo

Tu API pasó de "crear y listar" a manejar el ciclo de vida entero de un dato. Pongámosle nombre.

Definición técnica

CRUD reúne las cuatro operaciones básicas sobre cualquier dato persistente. Casi todo sistema de información —un banco, una red social, una tienda— se construye, en su base, sobre operaciones CRUD.

Cada letra = una operación = un verbo HTTP.
LetraOperaciónVerboTu endpoint
CCreatePOSTPOST /productos
RReadGETGET /productos y GET /productos/{id}
UUpdatePUTPUT /productos/{id}

💭 Lo que solemos pensar

"Por el verbo" o "por el método".

✓ La respuesta

Por el método HTTP, como vimos en REST. La misma dirección /productos/5 dispara acciones distintas según el verbo: GET la lee, PUT la actualiza, DELETE la borra. El recurso es el mismo; la acción la define el verbo. Pocas rutas, muchas acciones, todo predecible: eso hace que una API sea profesional.

Identidad (ID)

Parámetros de ruta

Parámetros de query

Filtros combinables por categoría y precio.

PUT

DELETE
