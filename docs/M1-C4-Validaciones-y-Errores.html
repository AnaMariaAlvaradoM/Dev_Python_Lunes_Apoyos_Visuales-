00 · FUNDAMENTO

🛡️ ¿Cómo se protegen los datos en Python?

Imagina esta escena en tu tienda: un empleado nuevo, con prisa, registra un producto sin nombre, con precio -5000 y stock -3. Le da guardar. Y tu API lo acepta feliz: responde 200 OK. Ese producto fantasma ya está en tu sistema.

Antes de arreglarlo, respondamos una pregunta que quizás ya te hiciste: si vienes de un lenguaje como Java, sabes que allí un dato se protege con private. ¿Dónde está el private de Python?

En Java: un muro obligatorio

En Java, cuando declaras un campo como private, el lenguaje impide a la fuerza que otro código lo toque desde fuera. Si lo intentas, el programa ni siquiera compila. El compilador es un guardia en la puerta. Para leer o cambiar ese dato pasas por métodos públicos, y también sueles crear DTOs (Data Transfer Objects): clases cuyo único trabajo es transportar datos entre las capas.

En Python: una señal, no un muro

En Python no existe private de verdad. Puedes tocar cualquier dato de cualquier objeto, siempre. No es un descuido: es una decisión de diseño. La filosofía de Python se resume en una frase de su comunidad: "aquí todos somos adultos responsables". Donde Java prohíbe, Python confía y señala.

Lo más parecido a private es un guion bajo antes del nombre:

solo ilustrativo 
self._precio    # "esto es interno, no me toques desde fuera"

Pero ese _ es una convención de caballeros, no una prohibición. Es un letrero de "PRIVADO" en una puerta sin cerradura: comunica la intención, pero no traba la puerta. En Java la puerta tiene cerradura; en Python tiene un letrero.

Si Python no puede impedir el acceso a los datos como Java, entonces su protección no puede estar en "esconderlos". ¿Dónde más podría estar?
▸ Ver explicación
En revisarlos. Como Python no protege escondiendo el dato, lo protege controlando qué entra. En vez de decir "nadie toca el precio", dice "el precio entra por una puerta donde reviso que sea válido". La protección se muda del acceso (Java) a la entrada (Python). Y esa puerta de entrada, en una API de FastAPI, se llama Pydantic.

Tu schema ES un DTO (con superpoderes)

Recuerda tus schemas de la clase pasada: ProductoCreate, CategoriaCreate. Cada uno es un objeto cuyo único trabajo es recibir y transportar los datos que entran a tu API. Eso es exactamente la definición de un DTO. Sin saberlo, ya estabas creando DTOs.

Pero el schema de Pydantic hace algo que el DTO de Java no hace gratis: valida al mismo tiempo que transporta. En Java necesitarías el DTO más una librería de validación aparte. En Python, Pydantic fusiona las dos cosas en una sola clase corta.

DTO (Data Transfer Object): objeto cuyo propósito es transportar datos entre las capas de una aplicación. En FastAPI, tus schemas de Pydantic cumplen ese rol y además validan cada dato que entra.

Es como el sobre de una carta con reglas impresas: "solo cabe una hoja tamaño carta, con remitente obligatorio". El sobre transporta el contenido y a la vez rechaza lo que no cumple.

La idea que ordena toda la clase

En Python los datos se protegen validando la entrada, no escondiendo el dato. Tu schema deja de ser un simple mensajero y se vuelve un guardián. Todo lo que sigue —Field, field_validator, model_validator— son formas de decirle a ese guardián qué debe revisar.

01 · VALIDACIÓN

🚧 Validar formato y límites con Field

Empecemos por el producto basura. Tu schema actual acepta cualquier cosa. Vamos a darle al guardián sus primeras reglas.

"El precio de un producto debe ser mayor que cero." ¿Eso es una regla del lenguaje Python, o una regla de tu negocio?
▸ Ver explicación
Es una regla de tu negocio. Python felizmente guarda -5000 en una variable; no le molesta. Eres tú, como dueño del sistema, quien decide que en tu tienda un precio negativo no tiene sentido. Y esas reglas se escriben en el schema, en la puerta de entrada, para que se apliquen solas cada vez que entra un dato.

Field: herramienta de Pydantic para agregar restricciones a un campo, más allá de su tipo. Le dice no solo "esto es un número", sino "es un número que debe cumplir estas condiciones".

Es la casilla de un formulario de papel bien diseñado: "Nombre (obligatorio, máx. 50 caracteres)", "Edad (número, mayor a 0)".

Las restricciones que usaremos

Estas son las reglas que le daremos a Field: ... significa obligatorio (el campo no puede faltar); gt=0 es mayor que cero; ge=0 es mayor o igual a cero; min_length y max_length fijan el largo mínimo y máximo de un texto.

El código

Este es el schema de producto completo. Reemplaza todo el contenido del archivo por esto:

app/schemas/producto_schema.py 
from pydantic import BaseModel, Field


class ProductoCreate(BaseModel):
    nombre: str = Field(..., min_length=3, max_length=50)
    precio: float = Field(..., gt=0)
    stock: int = Field(..., ge=0)
    categoria: str = Field(..., min_length=3)

Línea por línea: importamos Field junto a BaseModel. El nombre es texto obligatorio de entre 3 y 50 caracteres. El precio es decimal obligatorio y mayor que cero. El stock es entero obligatorio y mayor o igual a cero. La categoría es texto obligatorio de mínimo 3 caracteres.

Sin escribir un solo if, tu API ya rechaza el producto basura. Pydantic aplica estas reglas automáticamente cada vez que llega un producto. Así se protegen los datos en Python: no con private, sino con reglas en la puerta de entrada.

El precio debe ser mayor que cero (gt=0) y el stock mayor o igual a cero (ge=0). ¿Por qué el stock sí puede ser cero pero el precio no?
▸ Ver explicación
Porque un producto puede estar agotado (stock 0) pero seguir existiendo; en cambio, un producto que cuesta $0 normalmente es un error. Fíjate cómo la restricción técnica (gt vs ge) refleja una decisión de negocio. No elegiste gt=0 por capricho: lo elegiste porque entiendes el dominio. Esa es la diferencia entre copiar código y diseñarlo.

Pruébalo en Swagger

Levanta el servidor desde la carpeta tienda_dev (la que contiene a app), nunca desde dentro de app:

PowerShell · desde tienda_dev 
uvicorn app.main:app --reload

Abre http://127.0.0.1:8000/docs. En POST /productos, prueba primero con el producto basura (nombre vacío, precio -5000, stock -3): debe responder 422, listando cada campo que falló. Luego prueba con datos correctos: debe responder 200 OK.

El error 422, léelo bien

422 significa "Unprocessable Entity": entendí tu petición, pero los datos no pasan las reglas. Es un error del cliente (4xx), no del servidor. Y tú no lo escribiste: Pydantic lo genera solo. La validación vino gratis con declarar las reglas.

02 · VALIDACIÓN

🧹 Validar y limpiar con field_validator

Field es potente, pero solo entiende reglas estándar: largos, mayores, menores. No sabe de las reglas particulares de tu negocio, ni puede transformar un dato. Si alguien registra el nombre " teclado ", para Field eso tiene 12 caracteres válidos: pasa. Pero tú quieres guardarlo limpio.

field_validator: decorador de Pydantic para escribir tu propia función de validación sobre un campo. Puede revisar, rechazar (con ValueError) y transformar el dato.

Es el revisor de un formulario que no solo dice "esto está mal", sino que también corrige lo que puede: te tacha el espacio de más y deja el nombre bien escrito.

No es solo para espacios

field_validator sirve para cualquier regla propia sobre un campo: limpiar un texto, normalizar un formato (pasar a minúsculas), restringir a una lista de valores permitidos, validar un formato como un email, o rechazar contenido no deseado. Lo verás en varios usos distintos hoy.

La regla de oro

Un field_validator siempre debe devolver el valor (corregido o tal cual). Si no lo devuelves, el campo se guarda vacío. Es el error número uno de todo principiante con Pydantic.

Antes del código: ¿qué es cls?

En el código que sigue verás un parámetro llamado cls. No es una función ni una palabra reservada: es simplemente el primer parámetro que recibe un método de clase, y por convención se llama así (viene de class).

Compáralo con lo que ya conoces: en tus métodos normales, el primer parámetro es self, que representa la instancia (un objeto concreto, "este producto"). En un método marcado con @classmethod, el primer parámetro es cls, que representa la clase misma (el molde, no un producto concreto).

¿Por qué aquí es cls? Porque la validación ocurre mientras el objeto se está construyendo. En ese instante el producto todavía no existe como objeto terminado, así que no hay un self al cual apuntar; solo existe el molde. Por eso Pydantic valida a nivel de clase.

cls vs self: cls es la clase (el molde); self es la instancia (el objeto ya hecho). El decorador @classmethod obliga a que el primer parámetro sea cls.

El molde de galletas es la clase (cls): la forma con la que se hacen todas. Una galleta horneada es la instancia (self). La regla "ninguna galleta sale con menos de 3 cm" vive en el molde, porque se aplica antes de que la galleta exista.

En tus métodos anteriores el primer parámetro era self, pero en el validador es cls. ¿Qué representa cada uno y por qué el cambio?
▸ Ver explicación
No son lo mismo. self es la instancia (un producto concreto); cls es la clase (el molde). El cambio ocurre porque el @classmethod sobre el validador obliga a usar cls, y eso pasa porque la validación corre mientras el objeto se construye: todavía no hay instancia terminada, solo el molde. En la práctica casi nunca usas cls dentro del validador; está ahí por la sintaxis, igual que self aparece en tus métodos aunque a veces no lo toques.

El código: limpiar el nombre

Agregamos un validador que quita espacios sobrantes y rechaza nombres que sean solo espacios. Así queda el archivo:

app/schemas/producto_schema.py 
from pydantic import BaseModel, Field, field_validator


class ProductoCreate(BaseModel):
    nombre: str = Field(..., min_length=3, max_length=50)
    precio: float = Field(..., gt=0)
    stock: int = Field(..., ge=0)
    categoria: str = Field(..., min_length=3)

    @field_validator("nombre")
    @classmethod
    def limpiar_nombre(cls, valor: str) -> str:
        valor_limpio = valor.strip()
        if valor_limpio == "":
            raise ValueError("El nombre no puede ser solo espacios")
        return valor_limpio

Línea por línea: el decorador @field_validator("nombre") le dice a Pydantic que la función de abajo valida el campo nombre. El @classmethod es obligatorio en Pydantic v2 y va debajo del decorador anterior (el orden importa). Dentro, valor.strip() quita los espacios de los extremos; si tras limpiar quedó vacío, lanzamos un ValueError con nuestro mensaje (Pydantic lo convierte en 422); y al final devolvemos el nombre limpio, que es lo que se guardará.

Field ya pedía min_length=3. Si alguien manda tres espacios " ", ¿lo detiene el Field o el validador? ¿Por qué necesitamos los dos?
▸ Ver explicación
Ahí está la trampa: " " tiene 3 caracteres, así que min_length=3 lo deja pasar. Es nuestro field_validator quien lo atrapa, porque limpia primero y entonces ve que quedó vacío. Field y field_validator no compiten, se complementan: Field pone los límites gruesos, el validador afina lo que Field no puede ver. Un backend robusto usa capas de defensa, no una sola.

Segundo uso: restringir a una lista

Para ver que field_validator es mucho más que limpiar espacios, démosle otro uso: que la categoría solo pueda ser una de las que tu negocio maneja. Si tu tienda solo vende tecnología, papelería y hogar, no tiene sentido aceptar una categoría inventada. Agrega este validador dentro de la clase, debajo del anterior:

app/schemas/producto_schema.py · dentro de la clase 
    @field_validator("categoria")
    @classmethod
    def validar_categoria(cls, valor: str) -> str:
        categorias_validas = ["tecnologia", "papeleria", "hogar"]
        valor_limpio = valor.strip().lower()
        if valor_limpio not in categorias_validas:
            raise ValueError(f"Categoría inválida. Use una de: {categorias_validas}")
        return valor_limpio

Aquí valor.strip().lower() encadena dos transformaciones: limpia espacios y pasa a minúsculas, para que "Tecnologia", "TECNOLOGIA" y " tecnologia " se traten igual. Si la categoría limpia no está en la lista blanca, la rechazamos con un mensaje que además le dice al usuario qué opciones son válidas. Un buen error no solo dice "está mal": dice "esto es lo correcto".

Este validador normaliza (lower) y restringe (la lista) a la vez. ¿Notas que un mismo field_validator puede transformar y rechazar en la misma función?
▸ Ver explicación
Correcto. Un field_validator no está limitado a una sola tarea: puede limpiar, normalizar, comparar y rechazar, todo junto, siempre que al final devuelva el valor. Por eso es tan potente: es tu código, con toda la libertad de Python, corriendo dentro de la validación de Pydantic.

Pruébalo en Swagger

Manda un producto con nombre " Mouse inalámbrico ": responde 200 y el nombre queda limpio, sin espacios. Manda uno con "categoria": "juguetes": responde 422 con la lista de categorías permitidas. Y con "categoria": "TECNOLOGIA": responde 200 y la guarda en minúsculas. Validó y normalizó.

03 · VALIDACIÓN

🔗 Reglas entre campos con model_validator

Nuevo requerimiento: la tienda manejará ofertas. Un producto puede tener un precio de oferta, y hay una regla obvia: la oferta debe ser menor que el precio normal. Una "oferta" más cara que el precio real es un absurdo.

Aquí está el problema interesante: esta regla no la puede validar el precio solo, ni la oferta sola. Necesita ver los dos a la vez y compararlos.

model_validator: valida el modelo completo, con todos sus campos ya cargados, para poder comparar unos contra otros. Es el paso que subes cuando una regla depende de la relación entre varios campos.

Es como revisar un formulario de vuelo: la fecha de regreso no se valida sola, se valida contra la fecha de ida. Necesitas ver las dos casillas al tiempo.

No es solo para ofertas

model_validator sirve para cualquier regla entre campos: que una fecha de fin sea posterior a la de inicio, que si eliges "envío express" la dirección no esté vacía, que dos contraseñas coincidan, que un descuento no supere el total. Siempre que dos o más campos se relacionen, este es tu validador.

Usaremos mode="after": significa que el validador corre después de que Pydantic ya validó cada campo por separado. Cuando llega, ya sabemos que precio y oferta son números válidos; solo falta compararlos.

El código completo

Así queda el schema con todo integrado. Fíjate cómo conviven los validadores de campo con el de modelo:

app/schemas/producto_schema.py 
from pydantic import BaseModel, Field, field_validator, model_validator


class ProductoCreate(BaseModel):
    nombre: str = Field(..., min_length=3, max_length=50)
    precio: float = Field(..., gt=0)
    precio_oferta: float | None = Field(default=None, gt=0)
    stock: int = Field(..., ge=0)
    categoria: str = Field(..., min_length=3)

    @field_validator("nombre")
    @classmethod
    def limpiar_nombre(cls, valor: str) -> str:
        valor_limpio = valor.strip()
        if valor_limpio == "":
            raise ValueError("El nombre no puede ser solo espacios")
        return valor_limpio

    @field_validator("categoria")
    @classmethod
    def validar_categoria(cls, valor: str) -> str:
        categorias_validas = ["tecnologia", "papeleria", "hogar"]
        valor_limpio = valor.strip().lower()
        if valor_limpio not in categorias_validas:
            raise ValueError(f"Categoría inválida. Use una de: {categorias_validas}")
        return valor_limpio

    @model_validator(mode="after")
    def validar_oferta(self):
        if self.precio_oferta is not None and self.precio_oferta >= self.precio:
            raise ValueError("El precio de oferta debe ser menor al precio normal")
        return self

Lo nuevo: el campo precio_oferta es opcional. El float | None dice "es un decimal o puede ser nada"; el default=None lo hace omitible (es lo que permite no enviarlo). El validador validar_oferta usa mode="after" y recibe self, no cls: revisa que si hay oferta, no sea mayor o igual al precio, y devuelve el modelo completo (self), no un campo.

En el field_validator el primer parámetro era cls, pero en el model_validator es self. ¿Por qué el cambio?
▸ Ver explicación
No es capricho, tiene lógica. El field_validator corre mientras se construye el objeto: aún no hay instancia, así que trabaja con el molde (cls). El model_validator(mode="after") corre después de que el objeto ya está armado con todos sus campos: ya existe la instancia, así que recibe self y puede acceder a self.precio y self.precio_oferta. En resumen: cls cuando el objeto aún no existe, self cuando ya existe.

Pruébalo en Swagger

Un producto con precio 600000 y oferta 480000: responde 200, la oferta es menor. Con oferta 750000: responde 422, la oferta supera el precio. Y sin enviar oferta: responde 200, porque el campo es opcional. Confirmamos que nada de lo anterior se rompió.

04 · ERRORES

🏷️ Errores propios con excepciones personalizadas

Hasta aquí validamos lo que entra. Pero hay otro tipo de error: cuando el cliente pide algo que no existe, como GET /categorias/999 sin una categoría 999. Antes resolvíamos esto con un HTTPException escrito a mano en cada router. Funciona, pero tiene dos problemas.

Si tu API crece a 10 recursos y cada router repite el mismo raise HTTPException(404, "no encontrado"), ¿qué pasa el día que quieras cambiar el formato de ese error?
▸ Ver explicación
Tendrías que cambiarlo en los 10 lugares, y seguro olvidas uno. Ese es el primer problema: el manejo del error está repetido y disperso. El segundo es de lenguaje: HTTPException habla de HTTP dentro de tu lógica, cuando el router debería hablar de negocio. La solución profesional: crear una excepción propia con nombre de negocio y manejarla en un solo lugar central.

Excepción personalizada: una clase de error propia (que hereda de Exception) con un nombre que describe el problema en términos de tu negocio, no de HTTP.

Es la diferencia entre un empleado que grita "¡error 404!" y uno que dice "no encontré esa categoría". El segundo comunica el problema real; traducirlo al lenguaje técnico es trabajo de otra persona.

Paso 1 · El archivo de excepciones

Crea un archivo nuevo app/exceptions.py (clic derecho sobre la carpeta app, New File), al mismo nivel que main.py:

app/exceptions.py · archivo nuevo 
class CategoriaNoEncontrada(Exception):
    def __init__(self, categoria_id: int):
        self.categoria_id = categoria_id
        super().__init__(f"La categoría con id {categoria_id} no existe")

La clase hereda de Exception, la base de todos los errores en Python: al heredar de ella, nuestra clase es un error que se puede lanzar con raise. El constructor recibe el id que no se encontró, lo guarda, y le pasa a la clase padre el mensaje de error. Nota clave: esta excepción no menciona HTTP en ningún lado. Es un error puramente de negocio.

Paso 2 · El service lanza la excepción

En lugar de devolver None cuando no encuentra la categoría, el service ahora lanza tu excepción:

app/services/categoria_service.py 
from app.exceptions import CategoriaNoEncontrada

categorias = []


def obtener_categoria(categoria_id: int):
    for categoria in categorias:
        if categoria["id"] == categoria_id:
            return categoria
    raise CategoriaNoEncontrada(categoria_id)

Recorremos la lista; si encontramos la categoría, la devolvemos. Si el recorrido termina sin encontrarla, lanzamos la excepción. Ya no devolvemos None: declaramos el problema con nombre propio.

Antes el service devolvía None y el router lanzaba el 404. Ahora el service lanza la excepción. ¿No rompe esto la regla de que el service no sabe de HTTP?
▸ Ver explicación
Buena sospecha, pero la respuesta es no. Mira la excepción: no menciona HTTP en ningún lado, no dice 404, no sabe de la web. Es un error puramente de negocio: "esa categoría no existe". El service solo grita "¡no la encontré!" en su propio idioma. Quién traduce ese grito a un 404 es el manejador global. La capa de datos declara el problema; la capa web lo traduce. Las capas siguen intactas.

Paso 3 · El manejador global

Aquí está la magia: un solo lugar que atrapa todas las CategoriaNoEncontrada que se lancen en cualquier parte de la app y las convierte en una respuesta HTTP limpia. Así queda main.py:

app/main.py 
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse

from app.exceptions import CategoriaNoEncontrada
from app.routers import producto_router, categoria_router

app = FastAPI()


@app.exception_handler(CategoriaNoEncontrada)
def manejar_categoria_no_encontrada(request: Request, exc: CategoriaNoEncontrada):
    return JSONResponse(
        status_code=404,
        content={"error": str(exc)},
    )


app.include_router(producto_router.router)
app.include_router(categoria_router.router)

El corazón es la línea @app.exception_handler(CategoriaNoEncontrada): registra que cuando en cualquier lugar de la app se lance esa excepción, se ejecute la función de abajo, que devuelve una respuesta JSON con código 404 y el mensaje de la excepción.

Y ahora, el router: antes y después

Aquí se ve el cambio. Antes, el router tenía que revisar y lanzar el error él mismo:

ANTES · router con la lógica del error 
@router.get("/categorias/{categoria_id}")
def obtener(categoria_id: int):
    categoria = categoria_service.obtener_categoria(categoria_id)
    if categoria is None:
        raise HTTPException(status_code=404, detail="Categoría no encontrada")
    return categoria

Después, el router solo pide; si no existe, el service lanza y el manejador global responde:

DESPUÉS · router limpio 
@router.get("/categorias/{categoria_id}")
def obtener(categoria_id: int):
    return categoria_service.obtener_categoria(categoria_id)

El router pasó de 6 líneas a 3, y desapareció toda la lógica de error. Ya no necesita saber qué hacer cuando la categoría no existe: solo pide el dato. Su única responsabilidad volvió a ser conectar la petición con el service.

El router pasó de 6 líneas a 3, y no perdimos el manejo del error. ¿Dónde se fue ese manejo?
▸ Ver explicación
No se eliminó: se movió a un solo lugar central. Antes vivía repetido en cada router; ahora vive una vez en main.py. Si mañana quieres que todos los "no encontrado" incluyan la fecha o un código interno, tocas un solo archivo y cambia en toda la app. Eso es manejo centralizado, no disperso.

Pruébalo en Swagger

Haz GET /categorias/999 con un id que no exista: responde 404 con el mensaje "La categoría con id 999 no existe". Ese mensaje viajó desde tu excepción, sin que el router escribiera un solo raise. Todo lo manejó el handler central.

05 · ERRORES

📐 Que todos los errores hablen igual

Detente a mirar lo que tienes. Tu API responde errores de formas distintas: un error de validación (422) usa la estructura de Pydantic; tu error de categoría usa {"error": "..."}. Dos errores, dos formas. Para quien consume tu API, eso es una pesadilla: debe programar distinto para cada forma.

Si fueras quien desarrolla el front-end que consume esta API, ¿preferirías que cada error llegara con una forma distinta, o que todos llegaran con la misma estructura?
▸ Ver explicación
La misma, siempre. Si todos los errores tienen la misma forma (por ejemplo, siempre un campo error y un campo detalle), el front escribe una sola función para manejarlos y sirve para todos. Esto se llama respuestas consistentes, y es marca de API profesional. Una API que responde distinto en cada error es como un formulario donde cada campo pide la fecha en un formato diferente: funciona, pero es un caos de consumir.

Respuestas consistentes: todos los errores de la API comparten la misma estructura JSON, sin importar qué recurso o qué tipo de fallo los originó.

Es como una empresa donde todas las facturas tienen el mismo formato: sin importar el departamento, sabes exactamente dónde buscar el total, la fecha y el concepto.

Vamos a crear la última entidad de hoy, Proveedor, que nos deja ver dos cosas nuevas a la vez: un field_validator que valida un formato (un email) y el formato de error unificado.

Paso 1 · Schema de proveedor con validación de email

Crea el archivo nuevo app/schemas/proveedor_schema.py:

app/schemas/proveedor_schema.py · archivo nuevo 
from pydantic import BaseModel, Field, field_validator


class ProveedorCreate(BaseModel):
    nombre: str = Field(..., min_length=3, max_length=80)
    email: str = Field(..., min_length=5)

    @field_validator("email")
    @classmethod
    def validar_email(cls, valor: str) -> str:
        valor = valor.strip().lower()
        if "@" not in valor or "." not in valor:
            raise ValueError("El email no tiene un formato válido")
        return valor

Este es el cuarto uso distinto que le damos al field_validator hoy (limpiar, restringir a lista, y ahora validar un formato). Limpiamos y pasamos a minúsculas, revisamos que el email tenga una arroba y un punto, y si falta cualquiera lo rechazamos.

Nuestro validador solo revisa que el email tenga @ y punto. ¿Es una validación perfecta? ¿Aceptaría algo raro como a@b.c?
▸ Ver explicación
No es perfecta, y está bien que lo sepas: a@b.c pasaría. Validar emails al 100% es sorprendentemente difícil, y para eso existen librerías dedicadas. Hoy hacemos una validación básica a propósito, para que entiendas cómo se valida un formato con tus propias manos, sin cajas negras. En un proyecto real usarías una herramienta especializada, pero jamás le temerás, porque entiendes qué hace por dentro. Primero la mecánica, después la comodidad.

Paso 2 · Excepción del proveedor y formato unificado

Agrega esta segunda excepción en app/exceptions.py, debajo de la de categoría:

app/exceptions.py 
class ProveedorNoEncontrado(Exception):
    def __init__(self, proveedor_id: int):
        self.proveedor_id = proveedor_id
        super().__init__(f"El proveedor con id {proveedor_id} no existe")

Ahora, para lograr consistencia, actualizamos los manejadores en main.py para que ambos errores respondan con la misma estructura:

app/main.py · manejadores unificados 
from app.exceptions import CategoriaNoEncontrada, ProveedorNoEncontrado


@app.exception_handler(CategoriaNoEncontrada)
def manejar_categoria_no_encontrada(request: Request, exc: CategoriaNoEncontrada):
    return JSONResponse(
        status_code=404,
        content={"error": "recurso_no_encontrado", "detalle": str(exc)},
    )


@app.exception_handler(ProveedorNoEncontrado)
def manejar_proveedor_no_encontrado(request: Request, exc: ProveedorNoEncontrado):
    return JSONResponse(
        status_code=404,
        content={"error": "recurso_no_encontrado", "detalle": str(exc)},
    )

Definimos un formato fijo para todo error de "no encontrado": un campo error con un código y un campo detalle con el mensaje. Ambos manejadores devuelven la misma estructura: da igual si falló una categoría o un proveedor, el cliente recibe siempre la misma forma.

Los dos manejadores devuelven la misma estructura. Si mañana agregas 5 entidades más, ¿ves algún problema en repetir este bloque cada vez?
▸ Ver explicación
Sí, se repite mucho código, y detectar esa repetición es pensar como ingeniero. Hoy lo dejamos explícito (un manejador por entidad) porque estamos aprendiendo el patrón, y ver la repetición te hace sentir por qué molesta. Existe una forma de unificarlos usando una excepción base común, pero eso llega más adelante, cuando tenga sentido. Primero entender, después optimizar.

Pruébalo en Swagger

Manda un proveedor con email "correo-sin-arroba": responde 422 con "El email no tiene un formato válido". Con " Ventas@Andina.COM ": responde 200 y lo guarda como ventas@andina.com, sin espacios y en minúsculas. Y haz fallar dos 404 distintos (categoría 999 y proveedor 999): ambos responden con la misma estructura. Sin importar el recurso, el error habla el mismo idioma.

06 · CIERRE

🎓 Lo que lograste y lo que viene

Al empezar, tu API aceptaba un producto sin nombre, con precio negativo. Repasemos cuántas murallas tendría que atravesar hoy ese producto basura para entrar:

El fundamento. En Python se protege validando la entrada, no escondiendo datos. Tu schema es un DTO que además hace de guardián.

Field. Frena el precio negativo, el stock negativo y el nombre corto.

field_validator. Frena el nombre de solo espacios, restringe la categoría a una lista y limpia los datos.

model_validator. Frena ofertas ilógicas comparando campos entre sí.

Excepciones propias. Convierten los "no encontrado" en errores de negocio manejados en un solo lugar, y dejan los routers limpios.

Formato consistente. Hace que todos los errores hablen igual.

De una API ingenua que confiaba en todo, pasaste a una API que desconfía de todo dato que entra y responde con claridad cuando algo falla. Eso es calidad de backend.

El puente hacia adelante

Fíjate en algo pendiente: todas estas validaciones protegen los datos al entrar, pero los datos siguen viviendo en listas en memoria, que se borran al apagar el servidor. En el próximo módulo llega PostgreSQL, y esos datos por fin persisten en una base de datos real.

Y aquí está lo bonito: como pusiste toda la validación en los schemas, cuando lleguen las bases de datos tus reglas seguirán funcionando igual, porque viven en la capa de entrada, no en la de almacenamiento. Lo que blindaste hoy, te sirve mañana.
