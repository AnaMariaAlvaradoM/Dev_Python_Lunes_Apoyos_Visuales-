01 🔌

Conceptos fundamentales: la API

El "intermediario" que permite que una app móvil, una web y otros dispositivos hablen con un mismo servidor.

💼
Pregunta de negocio para abrir. Trabajas en una empresa de tecnología y deben construir una aplicación accesible desde una app móvil, un navegador web y otros dispositivos. ¿Cómo obtiene la app móvil los datos del servidor? La respuesta es: a través de una API. Por eso existe FastAPI.

¿Qué es una API?

API significa Application Programming Interface (Interfaz de Programación de Aplicaciones). Es un intermediario: recibe peticiones, las procesa y devuelve respuestas.

Ejemplo · vida real

Tienda física

Entras → hablas con el vendedor → el vendedor busca el producto → te lo devuelve. El vendedor es el intermediario.

Equivalente · técnico

Tienda online (con API)

La app envía una solicitud → la API busca en la base de datos → la API devuelve una respuesta en JSON.

Una petición entra, la API la procesa contra los datos, y una respuesta JSON regresa.
? Si trabajas en equipo y escribes una API con 100 endpoints, ¿cómo hace tu compañero para entender qué hace cada endpoint SIN leerte el código línea por línea?
Respuesta esperada ›
"Usando nombres claros y algún patrón."
Respuesta correcta ›
Exacto. REST proporciona ese patrón. Si ves DELETE /usuarios/5, automáticamente entiendes: "borra el usuario con ID 5". No hay sorpresas ni nombres raros. Es previsible.
02 🌐

HTTP, REST y JSON

El idioma de internet, las reglas que lo ordenan y el formato en que viajan los datos.

HTTP: el idioma de internet

Cuando abres google.com en el navegador, este habla HTTP con el servidor de Google. Una app móvil también habla HTTP. Ese idioma define verbos que indican qué quieres hacer:

GET
Obtener información · "Dame la lista de productos"
POST
Crear algo nuevo · "Crea un nuevo producto"
PUT
Actualizar lo existente · "Cambia el precio de este producto"
DELETE
Borrar algo · "Borra este producto"

REST: las reglas que lo ordenan

REST (Representational State Transfer) es una forma organizada de construir APIs sobre HTTP, para que sean predecibles.

Regla 1 — Cada recurso tiene una URL única.

/productos        # lista de todos los productos
/productos/5      # el producto con ID 5
/usuarios/10      # el usuario con ID 10

Regla 2 — Usas verbos HTTP para indicar la acción.

GET    /productos      # dame la lista
POST   /productos      # crea uno nuevo
DELETE /productos/5    # borra el producto 5

JSON: el formato de los datos

JSON (JavaScript Object Notation) es texto estructurado con llaves {} y pares clave-valor. Es como viajan los datos al enviar y recibir en una API.

ejemplo.json
{
  "nombre": "Laptop Dell",
  "precio": 1299.99,
  "stock": 10
}

Y un endpoint, ¿qué es?

Un endpoint es una URL específica de tu API que hace algo. http://miapi.com/productos es un endpoint; http://miapi.com/usuarios/5 es otro. Cada uno espera un método HTTP y devuelve datos o crea cambios.

03 ⚡

¿Por qué FastAPI?

Comparado con Django y Flask, y los tres superpoderes que lo hacen destacar.

Django

Maduro pero pesado

Existe desde 2005, con muchas librerías. Pero fue pensado para webs completas (HTML, formularios). Para solo una API es como comprar un auto entero cuando necesitas las ruedas. Más lento.

Flask

Ligero pero manual

Flexible y liviano, pero exige escribir mucho código a mano y no valida datos automáticamente. También más lento que FastAPI.

FastAPI · lo que usaremos

Moderno, rápido y automático

  • Moderno (2018 en adelante) y de los frameworks más rápidos del mundo.
  • Valida datos automáticamente — sin validación manual.
  • Genera documentación automática (Swagger).
  • Usa características modernas de Python.
? Si fueras director técnico y eligieras entre un framework que genera documentación automática vs. uno donde documentas todo a mano, ¿cuál elegirías y por qué?
Respuesta esperada ›
"El que genera documentación automática, porque ahorra tiempo."
Respuesta correcta ›
Exacto. Es un multiplicador de productividad: con documentación automática, el equipo frontend puede trabajar mientras el backend está en desarrollo. Sin eso, aparecen cuellos de botella.

Los 3 superpoderes

🍽️
Analogía de velocidad. Si Django es un mesero que atiende un cliente a la vez, FastAPI es un mesero que atiende a 1.000 clientes sin perder el ritmo. Eso lo logra gracias a ASGI (en lugar del WSGI de Django/Flask).
04 ⚙️

Instalación y configuración

Del Python instalado al servidor listo, paso a paso y sin asumir nada.

1 · Verifica que Python está instalado

Abre la terminal (PowerShell en Windows, Terminal en Mac) y escribe:

terminal
python --version

Debería mostrar algo como Python 3.12.3 (3.10 o superior). Si dice python is not recognized, instálalo desde python.org marcando "Add Python to PATH".

2 · Crea la carpeta del proyecto y entra

terminal
mkdir tienda_dev_senior
cd tienda_dev_senior

mkdir = make directory (crear). cd = change directory (entrar). Sabrás que estás dentro porque el nombre de la carpeta aparece en el prompt.

3 · Entorno virtual (venv): por qué

🧰
Si el Proyecto A necesita FastAPI 0.138 y el B necesita 0.100, instalar global genera conflicto. Un venv es una caja de herramientas aislada por proyecto.

4 · Crea y activa el venv

crear (Windows y Mac)
python -m venv venv
activar · Windows (PowerShell)
.\venv\Scripts\Activate.ps1
activar · Mac (Terminal)
source venv/bin/activate

Si funcionó, el prompt cambia y aparece (venv) al inicio. Ese prefijo indica que el entorno está activo.

⚠️
En Windows, si ves "cannot be loaded because running scripts is disabled", ejecuta esto una sola vez y vuelve a activar: 
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

5 · Instala FastAPI y Uvicorn

terminal · con (venv) activo
pip install fastapi uvicorn

pip es el instalador de paquetes de Python. fastapi es el framework; uvicorn es el servidor que lo ejecuta. Verifica con pip list que ambos aparezcan en la lista.

Estructura profesional del proyecto

Carpetas con propósito, listas para crecer. Cada una es un piso del edificio.

? Imagina una API con 100 endpoints, todos en un main.py de 5.000 líneas. Llega alguien nuevo y pregunta: "¿dónde cambio la validación de emails?". ¿Cuánto tarda en encontrarlo?
Respuesta esperada ›
"Mucho tiempo, se perdería."
Respuesta correcta ›
Exacto. Una estructura profesional separa responsabilidades. Es como un edificio: cada piso tiene un propósito. No metes la cocina en la habitación.

Las carpetas y qué guarda cada una

Hoy creamos toda la estructura; algunas carpetas se llenan en clases siguientes.

Las carpetas routers/ y services/ se llenan en la Clase 3; models/ en el Módulo 2 (base de datos); schemas/ lo estrenamos hoy con el endpoint POST.

¿Cómo se pronuncia "schemas"?

🗣️
Se lee aproximadamente "skímas" (la "sch" suena como "sk", no como "sh"). En singular, "schema" → "skíma". Significa "esquema": define qué datos espera recibir o devolver cada endpoint, y valida que lleguen correctos.

Los archivos __init__.py

En cada carpeta con código creamos un archivo __init__.py (puede ir vacío):

Windows (PowerShell)
ni app\__init__.py
ni app\routers\__init__.py
ni app\services\__init__.py
ni app\models\__init__.py
ni app\schemas\__init__.py
ni app\config\__init__.py
Mac (Terminal)
touch app/__init__.py
touch app/routers/__init__.py
touch app/services/__init__.py
touch app/models/__init__.py
touch app/schemas/__init__.py
touch app/config/__init__.py
🏢
¿Para qué un archivo vacío? Cuando una carpeta tiene __init__.py, Python la trata como un paquete: un lugar válido del que se puede importar código. Es como el letrero en la puerta de una oficina: la oficina puede estar vacía, pero el letrero avisa "este espacio es oficial, puedes entrar y buscar cosas aquí". Sin el letrero, Python pasa de largo.
🔎
Recuerda este síntoma: si más adelante ves ModuleNotFoundError: No module named 'app', casi siempre falta un __init__.py. Ya sabes dónde mirar.
06 🐍

Tu primer main.py

Tres endpoints GET, el servidor corriendo y JSON en el navegador.

En VS Code, crea el archivo app/main.py y escribe:

app/main.py
from fastapi import FastAPI

# Crear la aplicación FastAPI
# Los parámetros son metadata para la documentación automática
app = FastAPI(
    title="Tienda Dev Senior",
    description="API Backend profesional - Dev Senior Code",
    version="1.0.0"
)

@app.get("/saludo")
def obtener_saludo():
    return {
        "mensaje": "¡Hola desde FastAPI!",
        "estado": "activo"
    }

@app.get("/")
def raiz():
    return {
        "mensaje": "Bienvenido a Tienda Dev Senior",
        "version": "1.0.0"
    }

@app.get("/health")
def health_check():
    return {
        "status": "ok",
        "aplicacion": "Tienda Dev Senior"
    }

Leyéndolo línea por línea

  • from fastapi import FastAPI — importar es traer algo de una librería externa. FastAPI es una clase (un "molde") que representa una aplicación web.
  • app = FastAPI(...) — crear una instancia: un objeto basado en el molde. app es tu aplicación. title, description y version alimentan la documentación automática.
  • @app.get("/saludo") — un decorador: "cuando alguien haga GET a /saludo, ejecuta la función de abajo".
  • return {...} — devuelves un diccionario; FastAPI lo convierte solo a JSON.
💡
Nota honesta. Estos tres endpoints siempre devuelven lo mismo, y está bien: el trabajo de un health check es solo confirmar que la API está viva. Más adelante haremos uno que devuelve datos reales, creados por el usuario.

Ejecuta el servidor

terminal · con (venv) activo
uvicorn app.main:app --reload
  • app.main:app — "ve a la carpeta app, al archivo main.py, y busca la variable app".
  • --reload — recarga sola cada vez que cambias el código.

Cuando veas Uvicorn running on http://127.0.0.1:8000, tu API está corriendo. Pruébala en el navegador:

http://127.0.0.1:8000/saludo
http://127.0.0.1:8000/
http://127.0.0.1:8000/health
? ¿Dónde escribiste el código que convierte el diccionario de Python a JSON? ¿O lo hizo FastAPI automáticamente?
Respuesta esperada ›
"FastAPI lo hizo automáticamente."
Respuesta correcta ›
Exacto. Es un superpoder de FastAPI. En otros frameworks escribirías json.dumps() a mano. FastAPI lo hace solo porque sabe que construyes una API y todos los endpoints deben retornar JSON.
07 🧩

Recibir datos con POST y Pydantic

El salto clave: recibir datos del cliente, validarlos y guardarlos para leerlos después.

El método POST

GET obtiene información; POST crea información nueva. Con POST, el cliente no envía datos en la URL, sino en el body (cuerpo) de la petición, en JSON.

📝
De la vida real: GET es preguntar "¿qué productos hay?". POST es entregar un formulario lleno diciendo "registra este nuevo producto".

El problema que aparece de inmediato

Un producto debería llegar así... pero, ¿y si el cliente se equivoca?

correcto
{
  "nombre": "Laptop Dell",
  "precio": 1299.99,
  "stock": 15
}
roto
{
  "nombre": "Laptop Dell",
  "precio": "carísima"
}
# falta stock, precio es texto

Si guardamos eso, rompemos la app más adelante. Necesitamos un filtro que revise cada dato antes de dejarlo entrar. Ese filtro se llama Pydantic.

¿Qué es Pydantic?

🗣️
¿Cómo se pronuncia? Se lee aproximadamente "paidántik".
Definición · técnica

Validación en tiempo real

Librería de Python que usa las anotaciones de tipo (: int, : str) para revisar, en el momento en que llegan los datos, que cada campo tenga el tipo correcto y esté presente. Si algo no cuadra, rechaza la petición con un error claro. FastAPI la trae integrada.

Definición · vida real

El control del aeropuerto

Antes de subir al avión (tu lógica de negocio), cada pasajero pasa por seguridad: revisan que tenga todos los documentos (campos requeridos) y que sean válidos (tipos correctos). Sin papeles en regla, no pasa.

¿Quién es quién?

  • Pydantic = la librería de validación.
  • BaseModel = la clase base de Pydantic. El molde del que partimos.
  • schema = el esquema concreto que tú defines (ej. "un producto tiene nombre, precio y stock").

Paso 1 · Importar y definir el schema

Debajo de from fastapi import FastAPI agrega from pydantic import BaseModel. Luego, debajo de app = FastAPI(...) y antes de los endpoints:

app/main.py
from pydantic import BaseModel

# ==================
# SCHEMAS (validación con Pydantic)
# ==================

class ProductoCreate(BaseModel):
    nombre: str
    precio: float
    stock: int

ProductoCreate hereda de BaseModel (recibe sus superpoderes de validación). Declaramos tres campos con su tipo: nombre texto, precio decimal, stock entero. Si el cliente manda un precio como texto o se olvida del stock, Pydantic rechaza la petición — sin que escribamos una línea de validación.

Paso 2 · Una "memoria" temporal

⚠️
Esta lista vive solo mientras el servidor está encendido. Si lo apagas, los datos se pierden. Eso lo resolvemos en el Módulo 2 con una base de datos real (PostgreSQL). Hoy nos sirve perfecto para entender el flujo.
app/main.py
# "Memoria" temporal: aquí guardaremos los productos creados.
productos_db = []

Paso 3 · El endpoint POST

app/main.py
@app.post("/productos")
def crear_producto(producto: ProductoCreate):
    productos_db.append(producto)
    return {
        "mensaje": "Producto creado exitosamente",
        "producto": producto
    }

Usamos @app.post porque este endpoint crea. Al escribir producto: ProductoCreate, FastAPI toma el JSON del body, lo valida con Pydantic y nos lo entrega ya convertido. .append() lo agrega a la lista.

Paso 4 · El GET que lee datos reales

app/main.py
@app.get("/productos")
def listar_productos():
    return {
        "total": len(productos_db),
        "productos": productos_db
    }

Mismo recurso /productos, distinto verbo: si llega POST, ejecuta crear_producto; si llega GET, ejecuta listar_productos. Si no has creado nada, devuelve total: 0 y una lista vacía [] — y eso es correcto: no hay nada, así que no se muestra nada.

Nada aparece sin razón: el GET solo muestra lo que el POST guardó.
? Antes de crear ningún producto, llamas a GET /productos. ¿Qué crees que va a devolver?
Respuesta esperada ›
"Algún producto de ejemplo" o "un error".
Respuesta correcta ›
Devuelve una lista vacía (total: 0, productos: []). Tiene todo el sentido: no inventamos datos, así que si no has creado nada, no hay nada que mostrar. Todo lo que ves tiene una razón de existir.

Probar el flujo completo en Swagger

El POST necesita enviar un body, así que lo probamos en Swagger (/docs):

  1. Comprueba que está vacío: en GET /productos → "Try it out" → "Execute". Verás total: 0 y [].
  2. Crea un producto: en POST /productos → "Try it out", escribe el JSON de la Laptop Dell y "Execute".
  3. Vuelve a leer: regresa a GET /productos → "Execute". Ahora sí aparece tu Laptop Dell, porque el GET lee la lista real donde el POST la guardó.
? ¿Dónde validaste que el JSON tiene los 3 campos y que precio es un número? ¿Quién lo hizo?
Respuesta esperada ›
"No lo validé, FastAPI lo hizo."
Respuesta correcta ›
Exacto. El schema ProductoCreate se lo dijo a Pydantic y FastAPI lo aplicó solo. Pruébalo: borra stock o pon precio: "carísima" y ejecuta — verás un error 422 indicando qué campo está mal. Validación automática, cero código manual.
08 📖

Documentación automática: Swagger y ReDoc

Dos vistas de la misma API, generadas solas desde tu código. Nunca mienten.

/docs

Swagger · para probar

Más funcional: lista tus endpoints, muestra el schema ProductoCreate con sus campos, y trae botones "Try it out" para ejecutar peticiones en vivo. Es documentación y banco de pruebas.

http://127.0.0.1:8000/docs
/redoc

ReDoc · para leer

Más visual y limpio: presenta la documentación como un manual elegante, ideal para leer la API de un vistazo o compartirla con alguien que solo necesita consultarla.

http://127.0.0.1:8000/redoc
✨
Lo importante: ambas salen del mismo código, sin escribir documentación a mano. Usa Swagger cuando quieras probar, y ReDoc cuando quieras leer o presentar.
? Si eres frontend y necesitas consumir esta API, ¿qué prefieres: Swagger/ReDoc (generados solos y siempre actualizados) o un documento escrito a mano?
Respuesta esperada ›
"Swagger/ReDoc, porque puedo probar y leer directamente."
Respuesta correcta ›
Exacto. Un documento a mano queda desactualizado apenas el backend cambia algo. Swagger y ReDoc se regeneran solos con cada cambio del código, así que nunca mienten. Frontend y backend trabajan en paralelo sin esperar explicaciones verbales.
09 📌

El archivo requirements.txt

Para que cualquiera instale exactamente las mismas versiones que tú.

En la raíz del proyecto (no dentro de app/), crea requirements.txt:

requirements.txt
fastapi==0.138.0
uvicorn==0.49.0

Al instalar con pip install fastapi, Python toma la versión más nueva. En 6 meses, eso puede romper tu código. Fijando versiones (==0.138.0), todos instalan lo mismo. Quien clone el proyecto solo corre:

terminal
pip install -r requirements.txt
