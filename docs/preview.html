De la memoria a la base de datos

Comprenderás cómo una API conserva datos reales al conectar tienda_dev con PostgreSQL mediante SQLAlchemy.

Material exclusivo de clase

Este recurso es de autoría de Ana Alvarado y está destinado a estudiantes de la cohorte. No se autoriza su reproducción o redistribución fuera del contexto formativo.

01CONTEXTO

🎯 Punto de partida

Punto de partida

Tu API recibe 200 productos durante el día. El servidor se reinicia en la noche. ¿Qué debe ocurrir con esos datos al volver a encenderse?

Hasta ahora, tienda_dev ha guardado información en listas de Python. Esa solución permite practicar rutas, validaciones y lógica de negocio, pero no conserva los cambios cuando el proceso termina.

En esta clase transformarás esa arquitectura: los datos dejarán de depender de la memoria de la aplicación y pasarán a almacenarse en PostgreSQL.

Caso de negocio

Una tienda que no puede empezar de cero

Productos, categorías y proveedores forman parte de la operación del negocio. Perderlos por un reinicio implicaría reconstruir inventario, relaciones y disponibilidad. La persistencia convierte esa información en un activo estable del sistema.

El recorrido

Lo que construirás

Conectarás FastAPI con PostgreSQL, definirás tablas mediante modelos de SQLAlchemy, reemplazarás las listas en memoria por consultas reales y relacionarás Producto con Categoría y Proveedor.

Antes de comenzar

Necesitas PostgreSQL y pgAdmin instalados, además del proyecto tienda_dev construido durante el Módulo 1.

Antes de continuar

¿Por qué guardar datos en una lista funciona para practicar, pero no para operar un negocio?
Una lista existe dentro del proceso de Python. Permite manipular datos mientras la aplicación está encendida, pero desaparece al detenerse el proceso. Un negocio necesita almacenamiento independiente del ciclo de vida del servidor.

Antes de continuar

Imagina que tu tienda tienda_dev ya está en producción. Un cliente crea 200 productos durante el día. En la noche, el servidor se reinicia por una actualización. A la mañana siguiente, el dueño entra… ¿qué encuentra?
Una primera intuición podría ser: "Los 200 productos ahí, listos." Sin embargo, el criterio técnico es: Encuentra la lista vacía. Todo lo que se creó se perdió. Hoy los datos viven en una lista de Python (productos_db = []), y esa lista vive en la memoria RAM del programa. Cuando el programa se apaga, la memoria se borra. Ese es el problema que resolvemos hoy. No es un detalle técnico: es la diferencia entre un juguete y un sistema real.
03PROBLEMA

💾 El dolor: la memoria se borra

Qué problema resuelve

Primero comprobarás el límite de la solución actual: los datos creados durante la ejecución desaparecen cuando el proceso de Python termina.

Comprueba la pérdida de datos

  1. Levanta la API: uvicorn app.main:app --reload (desde tienda_dev, no desde app/).
  2. En Swagger (/docs), crea 2 o 3 productos nuevos con POST /productos.
  3. Confírmalos con GET /productos: ahí están.
  4. Detén el servidor (Ctrl + C) y vuélvelo a levantar.
  5. Haz GET /productos otra vez.

Qué van a ver: los productos que crearon desaparecieron. Solo quedan los 10 productos "de fábrica" que están escritos en el código (productos_db = [...]). Los que los datos agregaron en caliente, no.

La lección

Los datos en memoria son como escribir en un tablero: apenas apagas la luz (reinicias el programa), se borra. Necesitamos escribir en algo permanente: un cuaderno que siga ahí aunque cierres la oficina. Ese cuaderno es la base de datos.

Antes de continuar

¿Por qué los 10 productos de siempre sí sobreviven al reinicio, pero los que creamos en Swagger no?
Una primera intuición podría ser: "Porque los 10 están guardados y los otros no." Sin embargo, el criterio técnico es: Los 10 sobreviven porque están escritos en el código fuente (productos_db = [...]): cada vez que el programa arranca, Python vuelve a leer esa lista desde cero. Los productos nuevos solo se agregaron a la lista en memoria, nunca al código. Al reiniciar, la memoria se limpia y la lista vuelve a su estado original. Esto demuestra algo clave: la memoria no es almacenamiento. Es temporal por naturaleza.
04FUNDAMENTO

🗃️ Qué es una base de datos relacional

Qué problema resuelve

Ya sabemos que necesitamos algo permanente. Ahora entendemos qué es ese algo, antes de tocar código. Muchos escuchan "base de datos" por primera vez hoy: vamos desde cero.

Definición técnica

Una base de datos es un sistema organizado para almacenar, consultar y manipular información de forma permanente y eficiente. Una base de datos relacional organiza esa información en tablas que pueden relacionarse entre sí.

En palabras sencillas

Una base de datos es un sistema organizado para almacenar, consultar y manipular información de forma permanente y eficiente. Una base de datos relacional organiza esa información en tablas que pueden relacionarse entre sí.

Ejemplo real

Piensa en una hoja de cálculo de Excel, pero con superpoderes:

  • Cada hoja es una tabla (una para productos, otra para categorías).
  • Cada fila es un registro (un producto específico: "Laptop Dell").
  • Cada columna es un campo (nombre, precio, stock…).
  • Y lo relacional: una tabla puede apuntar a otra (un producto pertenece a una categoría).

Tres conceptos esenciales

Concepto Qué es En tienda_dev
Tabla Una colección de datos del mismo tipo La tabla productos
Fila (registro) Una entrada individual El producto "Mouse", id 2
Columna (campo) Una característica de cada entrada precio, stock, nombre

Qué es PostgreSQL

PostgreSQL es el programa que administra nuestra base de datos relacional. Es uno de los motores de bases de datos más usados y respetados de la industria: gratis, robusto y profesional. Ya lo instalaron de tarea. Es el "cuaderno permanente" donde almacenarás la información.

Qué es pgAdmin

pgAdmin es una interfaz visual para administrar PostgreSQL. En este recorrido lo usarás como visor para comprobar bases, tablas, columnas y registros. La estructura de las tablas se definirá desde el código.

Antes de continuar

Si PostgreSQL es el programa que guarda los datos, y tu API en Python es otro programa aparte… ¿cómo se hablan dos programas distintos que ni siquiera están escritos en el mismo lenguaje?
Una primera intuición podría ser: "Se conectan de alguna forma / con internet." Sin embargo, el criterio técnico es: Necesitan un traductor y un cable. PostgreSQL habla SQL; nuestra app habla Python. Para que se entiendan usamos dos piezas: un driver (psycopg, el cable físico que transporta los mensajes) y un ORM (SQLAlchemy, el traductor que convierte objetos de Python en SQL y viceversa). Eso es exactamente lo que montamos en el siguiente bloque.
05CONEXIÓN

🔌 El ORM y la conexión (database.py)

Qué problema resuelve

Aquí construimos el cimiento de todo el módulo: el archivo de conexión. Todo lo que a partir de hoy hable con PostgreSQL pasará por aquí. Es la pieza que reutilizaremos en cada clase que sigue.

Definición técnica

Un ORM (Object-Relational Mapping) es una capa que traduce entre objetos de Python y tablas de una base de datos relacional. En lugar de escribir SQL a mano, escribimos Python, y el ORM genera el SQL por nosotros.

En palabras sencillas

Un ORM (Object-Relational Mapping) es una capa que traduce entre objetos de Python y tablas de una base de datos relacional. En lugar de escribir SQL a mano, escribimos Python, y el ORM genera el SQL por nosotros.

Ejemplo real

Imagina que vas a un país donde no hablas el idioma. El ORM es tu intérprete: tú hablas en Python ("dame todos los productos"), el intérprete lo dice en SQL (SELECT * FROM productos), la base responde en SQL, y el intérprete te lo devuelve como objetos de Python que sí entiendes. Nunca hablas SQL directamente: el intérprete lo hace por ti.

Las tres piezas de la conexión

Antes de escribir código, para dejar claro quién es quién:

  • PostgreSQL → la base de datos (el cuaderno permanente).
  • psycopg → el driver: el cable que transporta los datos entre Python y PostgreSQL.
  • SQLAlchemy → el ORM: el traductor que centraliza todo y nos da una sola puerta de entrada, el engine.

Cómo imaginarlo

Analogía del tablero eléctrico: el driver (psycopg) es el enchufe que encaja en la pared. SQLAlchemy es el tablero eléctrico de la casa: centraliza la corriente y la reparte ordenada. Hoy instalamos el tablero y comprobamos que llega la luz.

Paso 2 — Crear el archivo de conexión

En una frase

Creamos el archivo que define cómo se conecta todo el proyecto a PostgreSQL.

Explicación a detalle

Este es el archivo más importante del módulo. Aquí definimos tres cosas: el engine (la puerta única a la base), el sessionmaker (la fábrica que produce sesiones de trabajo), y la Base (la clase madre de la que heredarán todos los modelos). De aquí en adelante, cada modelo y cada consulta parten de este archivo.

Cómo imaginarlo

Dónde va: crea un archivo nuevo llamado database.py dentro de app/config/. Esa carpeta la dejamos vacía en el Módulo 1 esperando justo este momento.

Antes del código, hay que entender la **cadena de conexión** (la URL). Léela por partes:
  • postgresql+psycopg → motor PostgreSQL usando el driver psycopg 3.
  • postgres:tu_contraseña → el usuario (postgres) y su contraseña.
  • localhost:5432 → dónde vive la base: tu propia máquina (localhost), puerto 5432 (el puerto por defecto de PostgreSQL).
  • tienda_dev → el nombre de la base de datos que usarás.

Código completo

📍 Archivo: app/config/database.py

Qué vas a hacer

Crear el módulo central de conexión.

Por qué lo hacemos

Toda la aplicación debe compartir una configuración única para el engine, las sesiones y la base declarativa.

Qué aprenderás

Cómo configurar SQLAlchemy sin mezclar la conexión con la lógica de negocio.

CREARapp/config/database.py
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, declarative_base


URL_CONEXION = "postgresql+psycopg://postgres:tu_contraseña@localhost:5432/tienda_dev"


engine = create_engine(URL_CONEXION)


SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)


Base = declarative_base()

Desglose línea por línea

  • from sqlalchemy import create_engine → traemos la función que crea el engine, la puerta a la base.
  • from sqlalchemy.orm import sessionmaker, declarative_base → traemos dos herramientas del ORM: sessionmaker (fábrica de sesiones) y declarative_base (la base para los modelos).
  • URL_CONEXION = "..." → la dirección completa de la base, en una sola línea. Es el "domicilio" de PostgreSQL.
  • engine = create_engine(URL_CONEXION) → crea el engine. Ojo: todavía no se conecta. Solo queda listo para cuando alguien lo use. Es como tener el número de teléfono guardado: aún no has llamado.
  • SessionLocal = sessionmaker(...) → crea una fábrica de sesiones. Cada vez que necesitemos hablar con la base, esta fábrica nos entrega una "sesión" nueva.
    • autocommit=False → no guardamos cambios automáticamente; nosotros decidimos cuándo con commit(). Más control, más seguro.
    • autoflush=False → no enviamos cambios a medias sin pedirlo. Evita sorpresas.
    • bind=engine → conectamos la fábrica al engine que creamos arriba.
  • Base = declarative_base() → crea la clase madre. Todos los modelos (Producto, Categoría, Proveedor) van a heredar de esta Base. Así SQLAlchemy sabe que son tablas.

Buena práctica

Cambia tu_contraseña por la contraseña real del usuario postgres que definieron al instalar PostgreSQL. Si la contraseña tiene símbolos como @ o /, ten presente que hay que escribirlos de forma especial.

Antes de continuar

Creamos el engine pero dije que 'todavía no se conecta'. ¿Por qué SQLAlchemy no se conecta de una vez al crear el engine?
Una primera intuición podría ser: "Para ahorrar / porque no se necesita aún." Sin embargo, el criterio técnico es: Por eficiencia. Abrir una conexión a una base de datos cuesta recursos. Si SQLAlchemy se conectara al arrancar y nadie usara la base, sería desperdicio. En vez de eso, el engine espera: se conecta solo cuando de verdad se necesita (cuando pedimos datos). Es "conexión perezosa" (lazy), y es una buena práctica: no gastes hasta que haga falta.

Paso 3 — Crear la dependencia get_db

En una frase

Creamos una función que entrega una sesión a quien la necesite y la cierra sola al terminar.

Explicación a detalle

Cada vez que un endpoint necesite hablar con la base, va a pedir una sesión de trabajo. Pero abrir una sesión y olvidar cerrarla causa fugas de conexión (se acumulan sesiones abiertas hasta que la base colapsa). Para evitarlo, creamos una función especial: abre la sesión, se la presta al endpoint, y garantiza que se cierre al final, pase lo que pase. FastAPI sabe usar este tipo de función a través de su sistema de inyección de dependencias (Depends).

Cómo imaginarlo

Dónde va: al final del mismo app/config/database.py.

Código

📍 Archivo: app/config/database.py (agregar al final)

Qué vas a hacer

Agregar la dependencia get_db.

Por qué lo hacemos

Cada petición necesita una sesión que se cierre incluso cuando ocurre un error.

Qué aprenderás

Cómo usar yield para administrar el ciclo de vida de una sesión.

MODIFICARapp/config/database.py
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

Desglose línea por línea

  • def get_db(): → definimos la función que provee sesiones.
  • db = SessionLocal() → pedimos una sesión nueva a la fábrica.
  • try: → empezamos un bloque protegido.
  • yield db → entregamos la sesión a quien la pidió (el endpoint). yield en vez de return es la clave: la función se pausa aquí, deja trabajar al endpoint, y retoma después para limpiar.
  • finally: → este bloque se ejecuta siempre, haya éxito o error.
  • db.close() → cerramos la sesión. Garantizado. Sin fugas.

Buena práctica

La conexión con lo que ya saben: Depends() es el mismo mecanismo que FastAPI usa para inyectar cosas en los endpoints. Aquí lo usamos para inyectar la sesión de base de datos. Lo van a ver en acción en los routers, más adelante en esta misma clase.

CREARprobar_conexion.py

06MODELO

🧱 El primer modelo: Producto (explicación total)

Qué problema resuelve

Este es el bloque más importante de la clase. Aquí aparece el concepto que más confunde a quien viene de otros lenguajes o ve bases de datos por primera vez: la diferencia entre un modelo y un schema. Lo explicamos a fondo con Producto. Las otras dos entidades solo lo repetirán.

El concepto que debe quedar claro: modelo ≠ schema

Ya tenemos producto_schemas.py desde el Módulo 1. Entonces, ¿por qué ahora creamos otro archivo para Producto en models/? ¿No es repetir?

No. Son dos cosas distintas con dos trabajos distintos:

Schema (Pydantic) Modelo (SQLAlchemy)
Vive en schemas/ models/
Hereda de BaseModel Base
Su trabajo Validar lo que entra y sale por la API Representar la tabla en la base de datos
Analogía El formulario que el cliente llena La ficha archivada en el cajón

Cómo imaginarlo

Analogía completa: el schema es el formulario que llenas en la recepción de un hospital (valida que pusiste bien tu edad, tu nombre). El modelo es la ficha médica que queda archivada en el sistema. Son parecidos, pero uno controla la entrada y el otro es el registro permanente. Necesitamos los dos.

Crear el modelo Producto

En una frase

Definimos, en código, la tabla productos con todas sus columnas y tipos.

Explicación a detalle

Un modelo de SQLAlchemy es una clase de Python que hereda de Base. Cada atributo de la clase se convierte en una columna de la tabla. SQLAlchemy lee esta clase y sabe exactamente qué tabla crear, con qué columnas y de qué tipo. Aquí definimos Producto con los mismos campos que ya manejaban en memoria: nombre, precio, stock, categoría — más el id que ahora generará la base sola.

Cómo imaginarlo

Dónde va: crea un archivo nuevo producto_model.py dentro de app/models/.

Código completo

📍 Archivo: app/models/producto_model.py

Qué vas a hacer

Definir la tabla productos como un modelo ORM.

Por qué lo hacemos

SQLAlchemy necesita una clase que describa el nombre de la tabla, sus columnas y restricciones.

Qué aprenderás

Cómo traducir una entidad de negocio a una tabla relacional.

CREARapp/models/producto_model.py
from sqlalchemy import Column, Integer, String, Float
from app.config.database import Base


class Producto(Base):
tablename = "productos"


id = Column(Integer, primary_key=True, index=True)
nombre = Column(String, nullable=False)
precio = Column(Float, nullable=False)
stock = Column(Integer, nullable=False)
categoria = Column(String, nullable=False)</code></pre>

Desglose línea por línea

  • from sqlalchemy import Column, Integer, String, Float → traemos las herramientas para definir columnas y sus tipos de dato.
  • from app.config.database import Base → traemos la clase madre que creamos en database.py. El modelo debe heredar de ella.
  • class Producto(Base): → definimos la clase Producto que hereda de Base. Esa herencia es lo que le dice a SQLAlchemy "esto es una tabla".
  • __tablename__ = "productos" → el nombre real de la tabla dentro de PostgreSQL. Convención: plural y en minúsculas.
  • id = Column(Integer, primary_key=True, index=True) → la columna id.
    • Integer → es un número entero.
    • primary_key=True → es la llave primaria: identifica de forma única a cada fila. La base la genera y la incrementa sola (1, 2, 3…). Aquí muere el contador_id y el global que usábamos en memoria.
    • index=True → crea un índice para buscar por id más rápido.
  • nombre = Column(String, nullable=False) → columna de texto. nullable=False significa obligatoria (no puede quedar vacía).
  • precio = Column(Float, nullable=False) → número con decimales, obligatorio.
  • stock = Column(Integer, nullable=False) → número entero, obligatorio.
  • categoria = Column(String, nullable=False) → texto, obligatorio.

Buena práctica

La base de datos genera el identificador al insertar cada fila. Por eso ya no necesitas contador_id ni variables globales para mantener una secuencia manual.

Antes de continuar

Ya teníamos producto_schemas.py desde el Módulo 1. ¿Por qué no borramos el schema y usamos solo este modelo nuevo? ¿Para qué queremos los dos?
Una primera intuición podría ser: "Porque el modelo ya tiene los campos, el schema sobra." Sin embargo, el criterio técnico es: Porque hacen trabajos distintos. El schema valida lo que entra por la API (que el precio sea positivo, que el nombre no esté vacío) y define lo que sale como respuesta. El modelo representa la tabla física. Si mezcláramos ambos, la validación de la API quedaría atada a la estructura de la base, y perderíamos flexibilidad: por ejemplo, el cliente no debe enviar el id (lo pone la base), pero sí debe recibirlo en la respuesta. Separar schema y modelo nos deja controlar entrada, salida y almacenamiento por separado. Es una de las buenas prácticas más importantes del backend profesional.

REEMPLAZARapp/services/producto_service.py
from sqlalchemy.orm import Session
from app.models.producto_model import Producto
from app.schemas.producto_schemas import ProductoCreate


def crear_producto(db: Session, datos: ProductoCreate):
nuevo = Producto(
nombre=datos.nombre,
precio=datos.precio,
stock=datos.stock,
categoria_id=datos.categoria_id,
proveedor_id=datos.proveedor_id,
)
db.add(nuevo)
db.commit()
db.refresh(nuevo)
return nuevo


def listar_productos(db: Session, categoria_id=None, precio_max=None):
consulta = db.query(Producto)
if categoria_id is not None:
consulta = consulta.filter(Producto.categoria_id == categoria_id)
if precio_max is not None:
consulta = consulta.filter(Producto.precio <= precio_max)
return consulta.all()


def obtener_producto(db: Session, producto_id: int):
return db.query(Producto).filter(Producto.id == producto_id).first()

Error frecuente

Nota de secuencia importante: este service usa datos.categoria_id y datos.proveedor_id. Eso obliga a actualizar el schema de entrada ProductoCreate: en el Módulo 1 tenía categoria: str (texto). Ahora debe tener categoria_id: int y proveedor_id: int | None = None, coherente con las llaves foráneas del Bloque 7. Lo ajustamos justo abajo. 📝 Decisión pedagógica (di esto en clase): el filtro de listar pasó de categoria (texto) a categoria_id (número). Es un cambio honesto que trae la relación: ahora filtras por el id real de la categoría. En la próxima clase, con los JOINs, podremos volver a filtrar por nombre de categoría cruzando tablas — mejor aún. Hoy sembramos, la próxima cosechamos.

Desglose de `crear_producto`

  • def crear_producto(db: Session, datos: ProductoCreate): → recibe la sesión (db) y los datos validados (datos). La sesión es nueva: antes no la teníamos.
  • nuevo = Producto(...) → creamos un objeto del modelo (no un diccionario). Le pasamos los campos desde datos. Fíjense: no ponemos id — la base lo genera.
  • db.add(nuevo) → marcamos el objeto para insertar. Todavía no está en la base: está "en cola".
  • db.commit() → confirmas. Aquí sí se escribe de verdad en PostgreSQL. Sin commit, no se guarda nada.
  • db.refresh(nuevo) → recargamos el objeto desde la base para traer el id que ella acaba de generar. Sin esto, nuevo.id estaría vacío.
  • return nuevo → devolvemos el producto ya con su id real.

Desglose de listar_productos

  • consulta = db.query(Producto) → arrancamos una consulta sobre la tabla de productos. Aún no traemos nada.
  • consulta = consulta.filter(Producto.categoria == categoria) → si mandaron categoría, filtramos. Compárenlo con el Módulo 1: antes era [p for p in productos_db if p["categoria"] == categoria]. Es la misma idea, pero ahora lo hace la base.
  • consulta = consulta.filter(Producto.precio <= precio_max) → segundo filtro combinable, igual que los filtros de la Clase 2.
  • return consulta.all() → .all() ejecuta la consulta y trae todas las filas que cumplen. Recién aquí se habla con la base.

Desglose de obtener_producto

  • db.query(Producto).filter(Producto.id == producto_id).first() → busca por id y trae el primero (o None si no existe). Reemplaza todo el for ... if ... return del Módulo 1 en una línea.

Buena práctica

La regla de oro de SQLAlchemy: los cambios que escriben (crear, editar, borrar) no son reales hasta el commit(). Las lecturas (.all(), .first()) se ejecutan al pedir el resultado. Grábense el commit: el olvido más común de la clase es agregar y no confirmar.

Antes de continuar

En crear_producto hacemos db.add(nuevo) y luego db.commit(). ¿Por qué dos pasos? ¿Por qué add no guarda de una vez?
Una primera intuición podría ser: "Es un paso de más / debería guardar directo." Sin embargo, el criterio técnico es: Porque separar add de commit permite agrupar varios cambios en una sola transacción. Imagina que necesitas crear un producto y descontar stock de otro a la vez: si algo falla a la mitad, quieres que nada se guarde. add prepara los cambios; commit los confirma todos juntos. Si algo sale mal antes del commit, se puede hacer rollback y deshacer todo. Es el mismo principio de un cajero: la transferencia no es real hasta que confirmas; si se cae la señal antes, no se movió el dinero.

MODIFICARapp/routers/producto_router.py
from fastapi import APIRouter, Depends
from sqlalchemy.orm import Session
from app.config.database import get_db
from app.schemas.producto_schemas import ProductoCreate, ProductoRespuesta
from app.services import producto_service


router = APIRouter()


@router.post("/productos", response_model=ProductoRespuesta)
def crear(datos: ProductoCreate, db: Session = Depends(get_db)):
return producto_service.crear_producto(db, datos)


@router.get("/productos", response_model=list[ProductoRespuesta])
def listar(categoria_id: int | None = None, precio_max: float | None = None,
db: Session = Depends(get_db)):
return producto_service.listar_productos(db, categoria_id, precio_max)


@router.get("/productos/{producto_id}", response_model=ProductoRespuesta)
def obtener(producto_id: int, db: Session = Depends(get_db)):
return producto_service.obtener_producto(db, producto_id)

Cómo leerlo

  • from app.config.database import get_db → traemos la dependencia que provee la sesión.
  • db: Session = Depends(get_db) → la línea clave, repetida en cada endpoint. FastAPI llama a get_db, obtiene la sesión, la inyecta como db, y la cierra al terminar. El endpoint no se preocupa de abrir ni cerrar nada.
  • response_model=ProductoRespuesta → la respuesta se arma con el schema de salida (con id). Gracias al from_attributes = True, FastAPI convierte el objeto SQLAlchemy a JSON solo.
  • producto_service.crear_producto(db, datos) → el router le pasa la sesión y los datos al service. El service hace el trabajo con la base.

Buena práctica

Continuidad: los filtros categoria y precio_max como query params son los mismos de la Clase 2. No cambió la interfaz para el cliente: cambió el motor por debajo. Esa es la belleza de tener capas bien separadas.

Antes de continuar

El router escribe db: Session = Depends(get_db) pero nunca llama a get_db() con paréntesis, ni cierra la sesión. ¿Quién abre y cierra la sesión entonces?
Una primera intuición podría ser: "El service / nosotros en el código." Sin embargo, el criterio técnico es: FastAPI, a través de Depends. Cuando llega una petición, FastAPI ve el Depends(get_db), ejecuta get_db hasta el yield (abre la sesión y la entrega), deja correr el endpoint, y cuando termina retoma get_db en el finally para cerrarla. Nosotros nunca abrimos ni cerramos a mano: el sistema de dependencias lo garantiza. Por eso get_db usaba yield y no return: para poder "volver" a cerrar después.

09RELACIONES

🔗 Relaciones: conectar las tablas

Qué problema resuelve

Tenemos tres tablas, pero aisladas. Un producto tiene una categoría escrita como texto ("tecnologia") y no está de verdad conectado a la tabla de categorías. Aquí aparece lo relacional de "base de datos relacional": conectamos las tablas con llaves foráneas.

El problema, en negocio

Hoy, en la tabla productos, la categoría es solo un texto suelto. Nada impide escribir "tecnولogia" mal escrito, o una categoría que no existe. Y si el dueño renombra una categoría, tendría que editar producto por producto. Eso no escala. La solución: que cada producto apunte a una fila real de la tabla categorias, por su id.

Definición técnica

Una llave foránea (foreign key) es una columna que apunta a la llave primaria de otra tabla. Crea una relación formal y verificada entre dos tablas: la base garantiza que el valor apuntado existe de verdad.

En palabras sencillas

Una llave foránea (foreign key) es una columna que apunta a la llave primaria de otra tabla. Crea una relación formal y verificada entre dos tablas: la base garantiza que el valor apuntado existe de verdad.

Ejemplo real

Es como el número de cédula en un formulario. En vez de reescribir todos tus datos en cada trámite, pones tu cédula, y esa cédula apunta a tu registro único en la base del gobierno. La llave foránea es esa cédula: un producto no repite los datos de su categoría, solo guarda el id que apunta a ella.

Relación 1:N (uno a muchos)

La relación que usamos hoy es uno a muchos: una categoría tiene muchos productos; un proveedor surte muchos productos. Del lado del producto, es "muchos a uno": muchos productos pertenecen a una categoría.

Qué vas a hacer

Usamos ForeignKey para declarar el vínculo, y relationship para poder navegar la relación cómodamente desde Python (por ejemplo, producto.categoria nos daría el objeto categoría completo).

Código

Qué vas a hacer

Agregar las llaves foráneas y relaciones de Producto.

Por qué lo hacemos

Qué aprenderás

Cómo usar ForeignKey y relationship para conectar modelos.

REEMPLAZARapp/models/producto_model.py
from sqlalchemy import Column, Integer, String, Float, ForeignKey
from sqlalchemy.orm import relationship
from app.config.database import Base


class Producto(Base):
tablename = "productos"


id = Column(Integer, primary_key=True, index=True)
nombre = Column(String, nullable=False)
precio = Column(Float, nullable=False)
stock = Column(Integer, nullable=False)

categoria_id = Column(Integer, ForeignKey(&quot;categorias.id&quot;), nullable=False)
proveedor_id = Column(Integer, ForeignKey(&quot;proveedores.id&quot;), nullable=True)

categoria = relationship(&quot;Categoria&quot;)
proveedor = relationship(&quot;Proveedor&quot;)</code></pre>

Desglose de lo nuevo

  • ForeignKey y relationship → dos importaciones nuevas de SQLAlchemy.
  • categoria_id = Column(Integer, ForeignKey("categorias.id"), nullable=False) → una columna entera que apunta a la columna id de la tabla categorias. nullable=False: todo producto debe tener categoría.
    • Fíjense: apunta a "categorias.id" (nombre de tabla + columna), no al modelo.
  • proveedor_id = Column(..., ForeignKey("proveedores.id"), nullable=True) → apunta a proveedores. nullable=True: el proveedor es opcional (un producto podría no tener proveedor asignado aún).
  • categoria = relationship("Categoria") → no es una columna. Es un "atajo" de Python: nos deja escribir producto.categoria y obtener el objeto Categoria completo, sin hacer la consulta a mano. SQLAlchemy la resuelve por nosotros.
  • proveedor = relationship("Proveedor") → lo mismo para proveedor.

Error frecuente

Importante para la clase: como cambiamos la columna categoria (texto) por categoria_id (llave foránea), la tabla vieja ya no sirve. En clase: bórrala en pgAdmin y deja que create_all la vuelva a crear con la nueva estructura. (Recuerden: create_all no modifica tablas existentes.)

Antes de continuar

Antes la categoría era un texto: \"tecnologia\". Ahora es un número que apunta a otra tabla: categoria_id = 3. ¿Qué ganamos al cambiar un texto legible por un número?
Una primera intuición podría ser: "Nada, el texto se entendía mejor." Sin embargo, el criterio técnico es: Ganamos integridad y mantenibilidad. Con el número apuntando a la tabla real: (1) la base impide asignar una categoría que no existe — no más "tecnولogia" mal escrito; (2) si el dueño renombra "tecnología" a "electrónica", cambia una fila en categorias y todos los productos reflejan el cambio, sin tocarlos uno por uno; (3) no repetimos el texto de la categoría en miles de productos. El número se ve menos legible, pero para eso está relationship: nos devuelve el objeto completo cuando lo necesitamos. Legibilidad para el humano, eficiencia para la base.

10VERIFICACIÓN

✅ Insertar y leer de verdad (el pago)

Qué problema resuelve

El momento que da sentido a todo. Vamos a insertar datos desde la API, verlos persistir, reiniciar el servidor, y comprobar que siguen ahí. El dolor del Bloque 1 se cura aquí.

Cómo imaginarlo

Recordatorio de decisión: insertamos y leemos desde el código (por la API). pgAdmin solo lo usamos para verificar que los datos llegaron. La base la escribe el código de la aplicación, no el mouse.

Orden de inserción (importa por las relaciones)

Como Producto apunta a Categoría y Proveedor con llaves foráneas, esos deben existir primero. No puedes asignarle a un producto la categoría 3 si la categoría 3 no existe aún.

Secuencia en Swagger (/docs):

  1. Crea una categoría — POST /categorias

    { "nombre": "Tecnología", "descripcion": "Equipos y electrónica" }

    Qué ves: respuesta 200 con la categoría creada, con su id (por ejemplo, 1). Ese id lo puso la base.

  2. Crea un proveedor — POST /proveedores

    { "nombre": "TechDistribuidora", "email": "ventas@techdist.com", "telefono": "3001234567" }

    Qué ves: 200 con el proveedor y su id (por ejemplo, 1).

  3. Crea un producto apuntando a esos ids — POST /productos

    { "nombre": "Laptop Dell", "precio": 1299.99, "stock": 15, "categoria_id": 1, "proveedor_id": 1 }

    Qué ves: 200 con el producto creado y su id. Ahora sí está en la base.

  4. Léelo — GET /productos Qué ves: una lista con tu producto real, traído de PostgreSQL.

La prueba de fuego

  1. Detén el servidor (Ctrl + C) y levántalo de nuevo.
  2. GET /productos otra vez.

Qué ven: el producto sigue ahí. No se borró. Eso es persistencia. Compárenlo con el Bloque 1, donde todo desaparecía. Ese contraste es la clase entera resumida en un momento.

Verificar en pgAdmin (como visor)

En pgAdmin: tienda_dev → Schemas → public → Tables → productos, clic derecho → View/Edit Data → All Rows. Ahí está la fila, con su categoria_id y proveedor_id apuntando a las otras tablas. Los datos existen fuera de Python. Son reales.

Antes de continuar

Reiniciamos el servidor y los productos siguieron ahí. En el Módulo 1 se borraban. No cambiamos el endpoint, ni el schema, ni la forma de pedir datos. ¿Qué cambió realmente?
Una primera intuición podría ser: "Cambiamos todo el código." Sin embargo, el criterio técnico es: Cambió dónde viven los datos. Antes vivían en la memoria del programa (se borra al apagar). Ahora viven en PostgreSQL (permanente). La API por fuera se ve casi igual — mismos endpoints, mismo Swagger — pero por dentro cambiamos el motor de almacenamiento. Eso demuestra el valor de la arquitectura por capas: pudimos cambiar el fondo (memoria → base de datos) sin romper la cara (los endpoints). Esa separación que aprendieron en la Clase 3 es justo lo que hizo esta migración posible.
